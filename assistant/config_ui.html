Personas — the fast-LM starter prompts, greetings & voices

Profiles — swappable bundles of memory + personalities (snapshot · fork · fresh)

A profile owns its own memory.db and personas.json (under config/profiles/<name>/); models, ports, tuning and voices stay global. Switch flips the active profile and restarts the cascade & research worker (any live conversation drops). Snapshot copies the current profile under the typed name without switching — to fork, snapshot then switch to the copy. The Memory and Personas tabs always act on the active profile.

Language models — each tier is a llama.cpp server: GGUF, GPU & runtime flags

Adding a model: drop the .gguf into Models/ (symlink that dir to wherever you keep weights), then Refresh and pick it.
GPU is the CUDA index under CUDA_DEVICE_ORDER=PCI_BUS_ID — here GPU 0 = 4090, GPU 1 = 3090. Live path (fast + embed) on GPU 0 with TTS; big LM alone on GPU 1.
Extra args are passed straight to llama-server (e.g. -ctk q8_0 -ctv q8_0 for a quantised KV cache). Any change here needs that tier's server restarted (./serve_fast_lm.sh etc).

Tuning — speech pacing, barge-in & noise rejection

Edits write to config.json; the cascade re-reads them on the next call, so reconnect (or start a new conversation) to hear changes. If sentence ends get clipped, raise Keep word-ending; if pauses feel long, lower Gap between sentences; if noise cuts the AI off, raise Barge-in frames.

Services — per-service output & restart (from the vinkona.sh tmux stack)

Stack mode …
Knowledge mode shuts down the fast LM, TTS, cascade & tunnel and runs a second big LM on the 4090 (port 11440) so the knowledge-host can split distillation across both GPUs. Applying does a full stack restart (~1 min); flip back to normal when done.
Vinur (knowledge host) …
The server-to-server research lane: /health says the host is there, the authed /drop handshake proves the token and reports drops held + open gaps. The same dot lives in the page header. Configure the link under knowledge_host in config.json (or the launcher's Connect wizard).

    
Logs are logs/<service>.log, written by vinkona.sh. “Restart”
      drops a request that the host-side monitor window picks up — so the stack must have been started with
      ./vinkona.sh start.

Self — Vinkona's personality, as she's defined it for herself

    

      
      
      
    

    
Two layers of self. Character traits are her structured identity
      (declared, always in the prompt) across depth layers — core (innate canon, locked:
      her grounding and effectively part of her values), compensated (characteristic
      adaptations: learned, situational ways a core trait gets expressed), surface (how
      she's being for now). An adaptation never replaces the core it grew from — it is always
      cast from it ("curious — expressed as patient and unhurried when he's debugging"), so the
      grounding stays present in what she enacts and revert to core always leads back.
      Adaptations need a core parent and a context, can't invert what they express, and can't
      touch values or boundaries — that's what keeps her able to disagree. Self-memories
      are the looser relational insights she writes over time. You're the owner here, so you can
      edit anything — including core canon (the conversation-only restriction exists to keep
      untrusted data out, not you). Changes apply on the cascade's next connection.

Research — Tier-3 background learning & the prompts that drive it

    

      

        
        
        
        
At session end the big LM proposes topics from the conversation; a worker reads them up and stores low-priority knowledge memories. Also enables the in-conversation “learn about X” tool.

A SearXNG instance (JSON API enabled). When Wikipedia has no article or rate-limits the worker, research falls back to web-search snippets — without this, those topics silently learn nothing (“no source” in the live feed).

      

      

        
        
        
        
Periodically pull snapshots from your Mac tools (calendar, RSS, files) and memorise them, so Vinkona has ambient awareness without checking live every time. Needs the tunnel/host up.

      

    


    

      Idle learning
      
When no session is open and the box has been quiet, the worker keeps learning on its own:
        it reflects on a window of past interactions (how they went, what to remember about you, what to research — and
        what it could now do better with current tools), researches a batch of topics, then consolidates world-knowledge
        (merge/split). Each cycle walks further back through history, then wraps to the newest — so old chats get
        re-examined, not the same recent ones repeatedly. All on the big LM — it stands down the instant you start talking.
